# OpenClaw Files 技术说明与答辩问答

## 1. 文档目的

这份文档用于整理 OpenClaw Files 当前版本的技术实现细节，方便在项目展示、课程答辩、比赛路演或评委问答环节中快速说明：

- 这个应用的整体技术架构是什么
- 前端、后端分别用了哪些语言，它们各自承担什么职责
- 为什么选择这些技术，而不是其他方案
- OpenClaw Gateway 是如何建立通讯、认证、配对并发送消息的
- 评委可能追问哪些技术问题，以及推荐如何回答

这份文档尽量贴合当前代码实现，而不是只写抽象设计。

---

## 2. 项目整体架构

### 2.1 一句话概括

OpenClaw Files 是一个“桌面端文件整理工作区”，采用的是：

`Electron 前端桌面壳 + Python 本地后端 + OpenClaw Gateway / Anthropic AI 分析`

它不是纯网页应用，也不是只有命令行脚本，而是把系统拆成了三层：

1. 桌面交互层
2. 本地业务执行层
3. AI 通讯与模型接入层

### 2.2 三层职责拆分

#### 第一层：桌面交互层

由 Electron + HTML/CSS/JavaScript 组成，负责：

- 创建桌面窗口
- 调用系统文件夹选择器
- 展示目录结构、分析结果、执行进度
- 让用户手动确认或筛选操作
- 调用本地后端接口

对应代码：

- `main.js`
- `index.html`
- `renderer.js`
- `styles.css`

#### 第二层：本地业务执行层

由 Python 后端组成，负责：

- 读取目录结构
- 生成给 AI 的目录摘要
- 调用 Gateway 或直连模型
- 解析模型返回的整理计划
- 校验操作路径与操作类型
- 真正执行文件移动、重命名、删除、创建目录
- 生成回滚信息并支持撤销

对应代码：

- `backend/server.py`
- `backend/file_analyzer.py`

#### 第三层：AI / Gateway 通讯层

负责：

- 与 OpenClaw Gateway 建立 WebSocket 连接
- 处理 challenge-response 认证
- 用设备私钥做签名
- 自动处理 pairing
- 保存 device token
- 通过 `chat.send` 发送 prompt
- 接收流式输出与最终结果

对应代码：

- `backend/gateway_client.py`

此外，云同步和定时任务能力由：

- `backend/cloud_sync.py`

负责，它通过 OpenClaw CLI 与 `bdpan` CLI 完成百度网盘上传和计划任务创建。

## 3. 前端、后端分别用了哪些语言，它们的作用是什么

### 3.1 JavaScript 的作用

JavaScript 在这个项目里出现了两个位置：Electron 主进程和渲染进程。

#### 3.1.1 在 Electron 主进程中的作用

主进程代码在 `main.js`，它负责：

- 创建窗口
- 管理应用生命周期
- 启动和关闭 Python 后端
- 检查后端是否准备就绪
- 调用系统原生对话框选择文件夹

主进程的几个关键点：

- `createWindow()`：创建应用窗口
- `ensureRuntimeConfig()`：准备运行时配置文件和状态目录
- `resolveBackendLaunch()`：判断当前是开发环境还是打包环境，并决定如何启动后端
- `startBackendServer()`：启动 Python 后端
- `waitForBackendReady()`：轮询检测本地后端是否可用

为什么这里用 JavaScript：

1. Electron 本身就是基于 Node.js 与 Chromium，JavaScript 是最自然的宿主语言。
2. 桌面窗口、IPC、系统对话框等功能在 Electron 里都有成熟 API。
3. 可以和渲染层共用语言，降低项目复杂度。

#### 3.1.2 在渲染进程中的作用

渲染进程代码主要在 `renderer.js`，它负责：

- 页面按钮和交互逻辑
- 当前目录、分析结果、执行状态的前端状态管理
- 调用本地 HTTP 接口
- 渲染目录树和操作列表
- 提供文件预览

例如：

- 选择目录后自动加载目录树
- 点击“分析”后调用 `/analyze`
- 点击“确认执行”后调用 `/execute`
- 点击“回滚”后调用 `/rollback`

为什么这里用 JavaScript：

1. 前端交互、状态更新、DOM 操作本来就最适合 JavaScript。
2. 便于快速迭代和调整界面表现。
3. 可以直接配合 HTML/CSS 构建桌面应用 UI。

### 3.2 HTML 的作用

HTML 负责页面结构，是界面骨架。

它主要定义：

- 顶部工具区
- 左侧目录浏览区
- 右侧分析与预览区
- 操作面板
- 状态与说明面板

为什么用 HTML：

1. Electron 本质上渲染的是网页界面。
2. HTML 能快速表达复杂信息布局。
3. 在比赛或答辩项目中，HTML 能以较低成本提供较高完成度的交互界面。

### 3.3 CSS 的作用

CSS 负责视觉呈现，主要包括：

- 布局
- 颜色和主题
- 卡片、标签、按钮样式
- 操作类型的视觉区分
- 预览区排版

为什么用 CSS：

1. 和 HTML/JavaScript 配合自然。
2. 能快速把工具型应用做出产品感和展示感。
3. 对答辩和展示来说，视觉完成度会直接影响评委第一印象。

### 3.4 Python 的作用

Python 是这个项目后端的核心语言，承担的是“真正处理业务”的工作。

主要职责有四类：

#### 3.4.1 本地 HTTP 服务

`backend/server.py` 使用标准库 `http.server` 起本地服务，负责接收前端请求并路由到后端功能。

目前主要接口包括：

- `/analyze`
- `/execute`
- `/rollback`
- `/cloud/status`
- `/cloud/upload`
- `/cloud/schedule`

#### 3.4.2 文件分析与计划生成

`backend/file_analyzer.py` 负责：

- 扫描目录结构
- 生成压缩后的 `folder_index` / `file_index`
- 构建分析 prompt
- 调用 Gateway 或直连模型
- 解析 AI 输出
- 修复非法 JSON
- 对计划做启发式增强

#### 3.4.3 文件执行与回滚

同样在 `backend/file_analyzer.py` 中，Python 还负责：

- 移动文件
- 重命名文件
- 重命名文件夹
- 创建文件夹
- 删除文件或目录
- 记录备份信息
- 回滚最近一批操作

#### 3.4.4 Gateway 与外部 CLI 集成

Python 负责：

- WebSocket 通讯
- Ed25519 设备签名
- 本地状态保存
- 调用 OpenClaw CLI
- 调用 bdpan CLI

为什么用 Python：

1. Python 做文件系统操作和路径处理效率高、代码可读性好。
2. AI 生态成熟，便于接入模型和处理结构化结果。
3. WebSocket、加密、命令行调用、打包分发都有成熟库支持。
4. 对原型开发和后续迭代都很友好。

### 3.5 JSON 的作用

JSON 不是主程序语言，但它是项目里最重要的数据格式之一。

它承担了三类任务：

1. 前后端通信格式  
   前端请求和后端响应都走 JSON。

2. AI 输出格式  
   模型返回的整理计划必须是严格 JSON，后端才能执行。

3. 本地状态存储格式  
   设备身份、device token 等状态都持久化为 JSON 文件。

为什么大量使用 JSON：

1. 前后端都容易处理。
2. 适合结构化表达 AI 输出。
3. 轻量、通用、便于调试。

### 3.6 PowerShell 的作用

PowerShell 不参与运行时业务，而是服务于构建与打包。

项目里的 `scripts/build-backend.ps1` 负责：

- 调用 PyInstaller
- 把 Python 后端打包成可分发目录
- 供 Electron 安装包集成

为什么用 PowerShell：

1. 目标平台主要是 Windows。
2. 适合在 Windows 环境中组织构建命令和路径。
3. 便于和 `npm run build:backend` 统一衔接。

## 4. 为什么要选择这些技术，而不是其他方案

### 4.1 为什么不用纯网页应用

如果是纯网页应用，会遇到几个问题：

- 无法自然访问本地文件系统
- 无法方便地做目录遍历和真实文件操作
- 无法像桌面应用一样安装、分发、使用

这个项目的核心任务是整理本地文件，所以必须具备桌面级能力。

### 4.2 为什么不用纯原生 C++ / C#

纯原生也可以做，但成本更高：

- UI 开发效率较低
- AI 集成和前端迭代不够快
- 跨平台成本更高
- 在课程项目和比赛项目中，开发周期不一定划算

Electron + Python 的组合更适合快速形成完整产品闭环。

### 4.3 为什么不是所有逻辑都写在前端

如果把文件操作和 AI 通讯全放在前端，会带来问题：

- 文件操作逻辑过重，渲染进程复杂
- 安全边界不清晰
- 与 Gateway、CLI、加密逻辑耦合太深
- 后续难以测试和维护

所以本项目把“展示”与“执行”分开：

- 前端负责展示和交互
- 后端负责业务和安全执行

### 4.4 为什么不是所有逻辑都写在后端

如果没有 Electron 前端，项目就会退化成命令行工具，不利于：

- 展示效果
- 用户交互
- 目录预览
- 操作筛选与确认
- 答辩时的可视化演示

因此当前方案是工程实用性和展示效果之间的平衡。

## 5. OpenClaw Gateway 通讯机制详解

这一部分是项目里最有“工程特色”的地方之一。当前项目不是简单地调用一个普通 HTTP 接口，而是实现了一个 OpenClaw Gateway WebSocket 客户端。

### 5.1 为什么使用 WebSocket，而不是 HTTP POST

根据当前项目文档和探测结果，本地 OpenClaw 实例主要暴露的是 Gateway WebSocket 协议，而不是传统的 `POST /v1/responses` 风格接口。

因此项目采用：

- WebSocket 建立连接
- RPC 风格发送方法调用
- 流式接收聊天事件

这种方式更适合：

- challenge-response 认证
- 流式结果返回
- 对话式会话管理
- Gateway 设备权限模型

### 5.2 GatewayClient 做了什么

`backend/gateway_client.py` 的 `GatewayClient` 负责整条链路：

1. 读取 Gateway 配置
2. 规范化 `ws://` 和 `http://` 地址
3. 加载或生成设备身份
4. 打开 WebSocket
5. 等待 `connect.challenge`
6. 用 Ed25519 私钥签名 challenge 相关载荷
7. 发送 `connect`
8. 必要时处理 pairing
9. 保存 `deviceToken`
10. 通过 `chat.send` 发送消息
11. 通过 `chat` 事件接收流式返回
12. 必要时通过 `chat.history` 获取兜底结果

### 5.3 第一步：读取配置

GatewayClient 初始化时会从环境变量读取：

- `USE_GATEWAY`
- `GATEWAY_URL`
- `GATEWAY_TOKEN`
- `GATEWAY_PASSWORD`
- `GATEWAY_AGENT_ID`
- `GATEWAY_USER`
- `GATEWAY_SESSION_KEY`
- `GATEWAY_SCOPES`
- `GATEWAY_STATE_DIR`
- `GATEWAY_AUTO_APPROVE_LOCAL_PAIRING`

这些参数决定了：

- 去连哪个 Gateway
- 用什么 token
- 以什么客户端身份连接
- 申请哪些权限
- 状态文件保存在哪里

### 5.4 第二步：准备设备身份

Gateway 采用设备身份机制，而不是只有一个简单 token。

客户端会先检查本地是否已经有设备身份文件：

- `.openclaw-state/identity/device.json`

如果没有，就自动生成一对 Ed25519 公私钥，并派生出：

- `deviceId`
- `publicKeyPem`
- `privateKeyPem`

`deviceId` 的生成方式是：

- 对公钥原始字节做 SHA-256

这样做的意义是：Gateway 可以稳定识别“当前请求来自哪一台设备”。

### 5.5 第三步：打开 WebSocket 连接

真正连接时，客户端调用：

- `_open_websocket()`

这一步会连到：

- `ws://127.0.0.1:18789`

然后调用：

- `_connect(ws)`

进入握手阶段。

### 5.6 第四步：等待 connect.challenge

Gateway 不会让客户端一连上就直接发业务消息。

服务端首先会发一个事件：

- `connect.challenge`

这个事件里带有：

- `nonce`

`nonce` 可以理解为一次性的随机挑战值，客户端必须用自己的设备身份去响应这个 challenge，证明自己不是伪造请求。

### 5.7 第五步：构造 connect 请求

拿到 `nonce` 后，客户端会构造一个 `connect` 请求，里面包括：

- 协议版本
- 客户端 ID
- 客户端版本
- 平台
- 模式
- 角色
- scopes
- capabilities
- device 信息
- auth 信息

其中最关键的是 `device` 和 `auth`。

#### 5.7.1 device 字段

`device` 字段包含：

- 设备 ID
- 设备公钥
- 签名
- 签名时间
- nonce

签名不是随便签一个字符串，而是把以下信息按 `v3` 规则拼接成 payload：

- device_id
- client_id
- client_mode
- role
- scopes
- signed_at_ms
- token
- nonce
- platform
- device_family

然后用本地 Ed25519 私钥对这个 payload 做签名。

#### 5.7.2 auth 字段

`auth` 字段里可能包含：

- `token`
- `password`

优先级大致是：

1. 如果配置了 `GATEWAY_TOKEN`，先用它
2. 否则尝试读取已保存的 `deviceToken`

### 5.8 第六步：处理 pairing

如果当前设备还没被 Gateway 授权，connect 过程中会返回类似：

- `pairing required`

本项目做了一个比较实用的自动化处理：

1. 识别错误是否是“需要 pairing”
2. 如果允许自动审批，就调用本地 OpenClaw CLI
3. 用 `devices list --json` 查出待审批设备
4. 找到当前 `deviceId` 对应的 request
5. 用 `devices approve <requestId> --json` 完成审批
6. 再自动重试一次 Gateway 操作

这样做的好处是：

- 用户首次接入时少手工操作
- 项目演示更流畅
- 设备授权逻辑仍然保留

### 5.9 第七步：保存 device token

当 connect 成功时，Gateway 返回的 payload 里可能包含：

- `deviceToken`

客户端会把它保存到：

- `.openclaw-state/identity/device-auth.json`

后续再次连接时，可以复用 device token，而不必完全依赖首次配置时的静态 token。

### 5.10 第八步：构造 sessionKey

在真正发消息前，客户端还要确定会话标识。

当前逻辑分两类：

#### 稳定会话 key

格式类似：

- `agent:main:main`

它适合表示某个稳定会话。

#### 请求级会话 key

真正发送业务消息时，会基于稳定会话 key 再附加一段随机 UUID，形成：

- `agent:main:main:req:<uuid>`

这样做的好处是：

- 每次请求隔离
- 减少串话
- 便于幂等处理和日志定位

### 5.11 第九步：发送 chat.send

真正发送消息时，调用：

- `send_message(prompt)`

内部会通过 WebSocket 发送一个 RPC 请求：

- `method = "chat.send"`

参数主要包括：

- `sessionKey`
- `message`
- `deliver = false`
- `idempotencyKey`

可以把这一步理解成：

“客户端把文件夹摘要 prompt 发给 OpenClaw，让它基于当前会话生成整理方案。”

### 5.12 第十步：接收流式聊天事件

发送后，Gateway 会通过 `event = "chat"` 不断回传数据。

客户端会关注几种状态：

- `delta`：流式增量文本
- `final`：最终完成结果
- `aborted`：中止结果
- `error`：错误结果

客户端会不断把增量文本拼起来，直到拿到最终结果。

### 5.13 第十一步：history 兜底

如果流式阶段没有拿到完整结果，客户端还会再调用：

- `chat.history`

从最近历史消息中找到最后一条 assistant 文本，用于兜底。

这个设计说明项目没有假设 Gateway 和流式过程永远完美，而是考虑了真实环境下的不稳定性。

### 5.14 第十二步：把返回文本送回 FileAnalyzer

Gateway 返回的是文本，最终回到：

- `FileAnalyzer._generate_text()`

随后：

1. 解析 JSON
2. 本地修复非法 JSON
3. 必要时请求模型修复 JSON
4. 归一化整理计划
5. 返回给前端显示

这就把 Gateway 通讯层和具体业务分析层连接起来了。

## 6. 用户点击“分析”后，系统内部到底发生了什么

这是展示时非常适合讲的一条完整链路。

### 6.1 前端发起请求

用户在界面里选择文件夹，点击“分析”。

前端会：

1. 读取当前目录
2. 收集用户额外输入的整理要求
3. 调用本地后端 `/analyze`

### 6.2 后端扫描目录

后端不会直接把整个目录无限制读进去，而是：

- 限制扫描深度
- 限制总文件数
- 限制单目录文件数
- 把原始目录压缩成 `folder_index` 和 `file_index`

这样做的目的是控制：

- 性能
- token 成本
- prompt 大小

### 6.3 后端构造 prompt

后端会把目录摘要、统计信息、采样信息和用户要求一起拼成 prompt。

如果目录特别大，还会自动降级 prompt profile，用更小的采样规模重试。

### 6.4 后端通过 Gateway 发消息

构造好的 prompt 交给 `GatewayClient.send_message()`，通过 WebSocket 发给 OpenClaw。

### 6.5 获得模型返回结果

模型返回一个整理计划文本，目标是严格 JSON，里面包含：

- `summary`
- `summary_points`
- `categories`
- `operations`

### 6.6 本地校验和修复

如果 JSON 不合法，系统会自动修复；
如果模型输出不够稳定，本地还会做归一化和启发式增强。

### 6.7 前端展示计划

前端把整理方案展示给用户，用户可以：

- 查看概述
- 查看每一步操作
- 关闭某类操作
- 单步确认
- 一次性执行全部剩余操作

### 6.8 用户确认后才真正执行

执行阶段不再依赖 AI，而是后端严格按照结构化操作列表执行本地文件改动，并保留回滚信息。

## 7. 项目当前实现中的工程亮点

### 7.1 AI 不直接碰磁盘

AI 只负责出计划，不直接执行系统命令。  
真正改文件的是本地后端。

这意味着：

- 可控
- 可审计
- 可回滚

### 7.2 有路径安全约束

所有操作路径都必须是相对路径，并且必须被限制在用户选择的根目录内，避免目录越界。

### 7.3 支持回滚

执行不是“一锤子买卖”。  
删除、移动、重命名等操作都保留必要的备份信息，可以撤销最近一批变更。

### 7.4 对模型不稳定性有兜底

不是假设模型永远输出正确 JSON，而是：

- 本地提取 JSON
- 本地清洗
- `json_repair`
- 再次请求模型修复 JSON

这是一种更工程化的思路。

### 7.5 有大目录控制机制

目录分析不是无边界扫描，而是有：

- 深度上限
- 文件数量上限
- prompt 抽样
- context overflow 重试

这能提升稳定性与成本可控性。

### 7.6 有桌面化分发能力

前端使用 Electron 打包，后端使用 PyInstaller 打包，再通过 `electron-builder` 打成安装包。  
这说明项目不只是开发机上的 demo，而具备实际交付能力。

## 8. 项目当前可以坦诚说明的局限性

在答辩时，适当承认局限性通常不会减分，反而能体现对工程边界的清醒认识。

### 8.1 Electron 安全还可进一步收敛

当前主窗口配置中：

- `nodeIntegration: true`
- `contextIsolation: false`

这在原型阶段便于快速开发，但正式产品化时，更推荐：

- 使用 `preload`
- 关闭 Node 直接暴露
- 最小权限化 IPC

### 8.2 token 使用量目前没有做界面化记账

当前可以估算每次整理成本，但还没有把每次请求的 token / 费用实时展示给用户。

### 8.3 模型选择目前更多依赖 Gateway 默认配置

当前 `send_message()` 接口没有把 `model`、`max_tokens` 显式透传给 `chat.send`，实际使用哪个模型通常取决于 Gateway 端配置。

### 8.4 目录分析是“摘要式”的，不是全文理解

优点是隐私和成本更可控，缺点是它主要依赖：

- 文件名
- 扩展名
- 路径结构
- 类型分组

而不是理解每个文件的完整正文内容。

## 9. 答辩中评委可能问到的技术问题与参考回答

下面给出一组高频问题。答题时不需要逐字照背，但可以参考这种表达逻辑。

### 问题 1：你的项目整体技术架构是什么？

参考回答：

这个项目采用的是 `Electron 前端桌面壳 + Python 本地后端 + OpenClaw Gateway` 的三层结构。  
Electron 负责窗口、目录选择和界面展示；Python 后端负责目录分析、文件操作、回滚和云同步；Gateway 层负责和 OpenClaw 进行 WebSocket 通讯与设备认证。  
这样拆分的好处是 UI、业务执行、AI 通讯各司其职，后续维护和扩展都更清晰。

### 问题 2：为什么要用 Electron？

参考回答：

因为我的应用核心是“整理本地文件”，它不是单纯网页工具。  
Electron 的优势是：

- 能快速获得桌面端分发能力
- 能调用系统文件夹选择器
- 前端开发效率高
- 适合做展示和产品化界面

对于比赛和课程项目来说，它能在开发速度和交付效果之间取得很好的平衡。

### 问题 3：为什么后端要用 Python？

参考回答：

Python 在这个项目里很适合做三件事：

- 本地文件系统操作
- AI 集成
- 外部 CLI 与 WebSocket 通讯

尤其是目录扫描、路径校验、回滚、JSON 修复、Gateway 认证这些逻辑，用 Python 写可读性和开发效率都比较高。

### 问题 4：AI 在这个系统里具体负责什么？

参考回答：

AI 不直接修改文件。  
它只负责根据目录摘要生成结构化整理计划，包括：

- 总体整理思路
- 分类建议
- 操作列表

真正执行这些操作的是本地后端，并且执行前还需要用户确认。所以它是“AI 负责分析，本地负责安全执行”的架构。

### 问题 5：如果 AI 给错了方案怎么办？

参考回答：

我做了多层保护：

1. 模型只能输出受限的操作类型
2. 所有路径必须是相对路径
3. 路径执行前会做根目录范围校验
4. 用户可以在界面里先看再确认
5. 执行后保留回滚信息，可以撤销最近一轮操作

所以它不是“AI 想怎么动文件就怎么动”，而是 AI 给建议，本地受控执行。

### 问题 6：回滚是怎么实现的？

参考回答：

每次执行文件操作时，系统都会记录足够的 `backup_info`。  
对于删除，会先备份到临时区域；对于移动、重命名、文件夹合并等，也会记录恢复所需的信息。  
回滚时按逆序恢复，所以能覆盖绝大多数实际整理场景。

### 问题 7：你的系统会不会泄露用户文件内容？

参考回答：

当前版本分析阶段主要上传的是目录结构摘要和文件元数据，而不是把所有文件内容全文发给模型。  
比如会发送：

- 相对路径
- 扩展名
- 类型分组
- 少量采样信息

这样可以在保证一定智能分析能力的同时，尽量降低隐私风险和 token 成本。

### 问题 8：分析成本和什么有关？

参考回答：

主要和：

- 文件数量
- 目录数量
- 层级复杂度
- 路径长度
- 用户额外输入的整理要求

有关，而不是和文件大小线性相关。  
因为当前分析主要看目录摘要，而不是把大文件内容全文送给模型。

### 问题 9：如果目录特别大怎么办？

参考回答：

我做了边界控制：

- 有扫描深度上限
- 有总文件数上限
- 有单目录文件数上限
- 有 prompt 抽样策略
- 如果 prompt 太大，会自动切换更小的 profile 重试

所以它不会随着目录规模无限膨胀。

### 问题 10：为什么不是直接用规则引擎？

参考回答：

规则引擎适合处理规则明确、场景稳定的整理任务，但真实文件夹往往存在：

- 命名混乱
- 类型混放
- 半结构化语义

AI 的价值在于能做更强的语义归类和整体结构建议。  
同时我又用本地规则来约束执行，所以这是“AI 负责理解，规则负责安全”。

### 问题 11：OpenClaw Gateway 是怎么连接的？

参考回答：

连接流程大致是：

1. 读取 Gateway 配置
2. 加载或生成本地设备身份
3. 建立 WebSocket
4. 等待 `connect.challenge`
5. 用本地 Ed25519 私钥对 challenge 相关载荷签名
6. 发送 `connect`
7. 如果需要，自动用 OpenClaw CLI 完成 pairing
8. 保存 Gateway 返回的 device token
9. 用 `chat.send` 发送分析 prompt
10. 通过流式 chat event 接收结果

### 问题 12：你为什么不用普通 HTTP 接模型？

参考回答：

因为当前 OpenClaw 实例暴露的是 Gateway WebSocket 协议。  
另外，WebSocket 方案更适合：

- challenge-response 认证
- 设备配对
- 流式响应
- 会话化管理

对这个项目来说，比单纯 HTTP POST 更匹配 OpenClaw 的接入方式。

### 问题 13：这是不是只是个 demo？

参考回答：

我认为它已经超过“概念 demo”的阶段了。  
因为它具备几个完整闭环：

- 真实目录分析
- 结构化操作计划
- 本地执行
- 回滚
- 云同步
- Windows 安装包分发

当然，从正式产品角度看，Electron 安全收敛、操作审计、token 计量等地方还可以继续优化。

### 问题 14：你觉得当前系统最大的工程短板是什么？

参考回答：

我认为当前最值得继续优化的是两个方向：

1. Electron 安全收敛  
   例如用 preload 替代当前较宽松的渲染进程权限。

2. 可观测性  
   例如把每次分析的耗时、token、失败原因、回滚记录做得更透明。

这样回答既诚实，也体现了对工程成熟度的判断。

### 问题 15：下一步准备怎么优化？

参考回答：

我会优先做三件事：

1. 增加每次整理的 token / 成本可视化
2. 收紧 Electron 安全边界
3. 提高模型输出稳定性和结构化约束

如果继续产品化，还可以进一步支持：

- 多云盘同步
- 更丰富的文件内容理解
- 可配置整理模板
